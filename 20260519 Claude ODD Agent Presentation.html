The Business Problem

What does an ODD analyst do?

  • Read a completed manager DDQ (100+ pages)
  • Cross-reference 25+ appendices (SOC reports, annual reports, policies, org charts)
  • Write 22 topic assessments across 4 chapters
  • Assign risk ratings per topic and per chapter
  • Produce a formal ODD report for the investment committee

The cost

Each ODD report takes days of analyst time, much of it reading, cross-referencing, and writing structured prose that follows a repeatable pattern.

The agent aims to produce a first draft that reduces rework from days to hours.

Agent Evaluation — RAG vs Human ODD Report

Two jobs, scored separately [judge:GPT5.5 deepthinking]

Benchmarked against the human ODD analyst report. The narrative (facts) and the risk ratings score very differently — and fail for different reasons.

1. Narrative — facts 6.5 / 10

Human analyst benchmark: ~9 / 10

Strengths: strong factual extraction from DDQ & appendices; good structural coverage across ODD topics.

Weaknesses: limited mandate-specific focus (APG context often missing); generic firm-level content; verbose; lacks judgment from meetings & analyst experience.

Usable as a draft, but needs refinement for relevance and clarity.

2. Risk assessment — ratings 3 / 10

Human analyst benchmark: ~9 / 10

Findings: overestimates risk by relying on generic disclosures; poor calibration to mandate-specific residual risk; internal inconsistencies between rationale and rating.

Technical Solution

Source hierarchy — classified at ingest

Every document gets a source_role at ingest; the role maps to a numeric source_tier. Lower tier = higher-priority evidence. The hierarchy is data, not code.

# _config.py SOURCE_TIER_BY_ROLE = { "manager_completed_ddq": 0, "appendix": 1, "policy": 2, "assurance_report": 3, "public_disclosure": 4, "other": 5, "report_template": 9, }

Keyword overrides

SOURCE_TIER_RULES promotes a document by filename keyword: policy / compliance / mnpi → 2, soc / isae / assurance → 3, annual / 10-k → 4.

Excluded from the index

source_role='report_template' (tier 9) — the ODD template must never be retrieved as evidence.

Tier flows document → chunk → Vector Search index, and drives reranker priority — balanced quotas per source class so tier-0 DDQ chunks can't crowd out a required appendix.

Technical Solution

Five-pass retrieval — same question, different evidence shelves

For each ODD topic, the agent builds one search brief: section code + topic title + chapter title + topic prompt in agent.md + matching appendix titles. It then checks the DDQ first, followed by supporting files.

Worked example — A0300 Financial condition. The topic prompt in agent.md tells the search what an ODD analyst would look for:

"A0300 Financial condition of the firm · Firm & Organisation · check annual report, audited financials, credit-rating evidence, and DDQ; extract AUM/revenue, net income/EBITDA, profitability, liquidity/funding, fee revenue vs fixed costs, concentration · Appendix 8 – 2024 Annual Report – State Street Corporation.pdf"

Evidence passWhat it means for the analystSource searched
1. Matching DDQ sectionStart with the manager's answer for this exact topic.source_role=manager_completed_ddq + A0300
2. Referenced appendicesIf the DDQ answer mentions appendices, search those supporting files.source_role=appendix
3. Prompt-required appendixIf the topic prompt says to check a source, add the matched filename to the search.source_role=appendix
4. DDQ fallbackIf that section is thin, scan the rest of the DDQ for related facts.source_role=manager_completed_ddq
5. Unlabelled evidenceFor sources without DDQ section codes, match by topic meaning.DDQ, appendix, policy, assurance, disclosure, other

Here source_role means document type, not a person's role. Prompt-to-file examples: “annual report” → Appendix 8, “ownership structure” → Appendix 7, “AML/KYC/sanctions” → Appendix 15.

Technical Solution

Externalized agent spec: agent.md

All prompt content lives in a single editable file, not in notebook code.

  • §1 General prompt — tone, length, structure, citation format
  • §2 Risk ratings — 4-level scale with definitions
  • §3 Topics — 22 DDQ topics, each with optional Prompt:
  • §4 Part 2 — chapter rollup rules
  • §5 Output contract — JSON schema the code expects
## 3. Part 1 — DDQ Topics - A0300 Financial condition Prompt: do not rely only on the DDQ. Always check the latest annual report, audited financials, credit-rating evidence. Extract: revenue, net income, profit margin, EPS, mgmt-fee revenue vs fixed costs … - A0900 Key person risk Prompt: key person risk, succession and retention planning.

The analyst can tune behavior by editing agent.md — no notebook changes, no redeployment. Topic prompts drive both retrieval queries and assessment generation.

Lessons Learned — Fix 1

Evidence quality: the question leakage bug

Before

  • DDQ question wording was mixed into chunk_text
  • Assessment model quoted the question as if it were the manager's answer
  • All sources treated equally — no tier metadata on chunks
  • Appendix metadata went stale when classification changed but chunks weren't rebuilt

After

  • Split embedding_text (question+answer, for retrieval) from chunk_text (answer only, for evidence)
  • Assessment model sees only manager answers and supporting evidence
  • Source tier and role flow from document → chunk → Vector Search index
  • Metadata changes trigger automatic status reset and rebuild

Key insight: In RAG, what you embed for retrieval and what you show to the LLM as evidence are two different concerns. Mixing them caused the model to treat the questionnaire prompt as a factual source.

Lessons Learned — Fix 2

Retrieval precision: the A0300 case study

Problem: A0300 (Financial Condition) should always check the annual report. Appendix 8 contained revenue, net income, EPS, and margins — but the agent said "no relevant financial data available."

Root cause: The right evidence was in the index, but never reached the reranker. DDQ tier-0 chunks crowded it out of the top-N candidate list.

Three generic fixes

  • Prompt-to-appendix bridging — topic prompt says "annual report", matched to Appendix 8 filename automatically
  • Balanced reranker sampling — reranker sees candidates from 4 buckets (section, semantic, appendix, mandatory external)
  • Over-fetch before filtering — more candidates before final selection, so useful pages aren't lost behind decoys

Key insight: In RAG with tiered sources, the highest-tier evidence can starve other required sources. Retrieval needs explicit slots per source class, not just a single ranked list.

Technical Solution — Work in Progress

Pointing back to the exact source

Every claim should cite a checkable location — a PDF page, or a Word paragraph / page — so an analyst can verify it in seconds.

What works today

  • Unified provenance contract — parsed rows carry source_page_num, source_para_num, locator type and label; chunks carry start/end ranges
  • PDF appendices cite real page numbers
  • DDQ / DOCX cite paragraph-based locators
  • Citations render as document | locator | tier — internal counters (E1, E2) no longer leak as if they were references

Not solved yet

  • DOCX has no rendered Word page number — source_page_num is null, so citations fall back to paragraph locators instead of the page a reviewer sees
  • Needs a Word page-mapping step upstream in 03_parse_sources
  • Reviewers want Appendix | page | quote | confidence — the pipeline is not fully there

This is the weakest part of the pipeline today. Provenance unification is in place, but reliable page-level citation for Word documents still needs more work.
